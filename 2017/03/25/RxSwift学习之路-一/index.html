RxSwift学习之路(一)

Posted on 2017-03-25 | Edited on 2017-03-26 | In iOS

开始

从标题来看, 我又挖了一个坑, 之前的AFNetworking的坑还没有填好, 好惭愧, 后面回补. 事情的起因是这样的, 在过年在家的时候, 看到一个介绍RxSwift的文章, 然后介绍MVP, MVVM之类之类的, 反思自己私下玩的花生iOS端的重构

第一次重构

第一次重构, 感觉只是把之前工程师所写的, 按照自己的逻辑和代码规范重新写了一遍, 对于其中的一些功能, 的确做到了更加面向对象, 但随着自己不断的学习和反思, 感觉这不是重构的终点站, 而是另一个航程的开始而已, 于是, 开始了第二次重构,

第二次重构

这次重构, 将业务逻辑更加抽象, 大量使用 泛型+协议, 使得 “从服务器取数据集” -> “转换模型” -> “映射到cell展示” 这种模式, 只需要做一些配置即可, 目的在于工程师只需要把精力放到那些不同的地方即可, 例如cell的布局, 模型的内容, 以及模型和cell数据填充和显示的关系. 直到我读到那个MVVP, MVP之类的文章后,

第三次重构

我开始反思整个项目的结构, 并没有一个清晰的, 统一的设计, 之前所做的, 无非是让ViewController不那么臃肿, 分离出来一个Manager, 但是Manager和外面的交流, 各种代理和回调, 使得局面混乱, 至少给阅读上增加了难度, 于是, 我看到的RxSwift带给我的第一个好处, 我要用Observable尽可能多的取代回调和代理. 怀着这样一个初衷, 我开始了我的Rxswift之路. 当然, 我知道, 这不是最后一次重构, 因为重构是我对以前自我的怀疑和审视, 也是一种自我的成长.

入门

Rx系列的内容对我来说, 是一种思维方式, 而人的思维方式是比较难转变的, 所以, 这种东西的学习成本也是很高的. 所以脑海中不要浮现 21天精通xxx, 本文没有那么高的造诣, 只有简单的使用心得和总结.

正文

首先, 对于Rx这一系列的编程, 而弱水三千只取一瓢饮, 我用我相对熟悉的swift来进行归纳和总结, 我的直观感受就是, 他们是事件的流, 从一条流, 一条流的变形, 再到多条流的混合, 以及一些复杂的情况(暂时还没遇到), 从阅读上, 更加的线性. 首先祭出神器, 没图你说个JB, 当然手机上也可以下载RXMarbles应用来随时随地学习.

基础概念

这里是根据我使用过的一些功能进行讲述, 而并不会穷举所有的基础概念, 工程量偏大, 大家可以参考我引用栏里的内容, 进行查看, 这里就不赘述了. 人说脱离业务说架构等于耍流氓, 这里我们也从一个例子入手.

基础

首先, 要理解事件, 比如一次网络请求, 由一个按钮触发, 然后传递到网络上, 返回结果, 处理事件(onNext). 一次事件结束. 这里的结束有两种可能, 一种是正确的结束(onComplete), 一种是错误的结束.(onError)

思路

将当前类想象为一座信号塔, 会向外发射微博登录成功或者失败的消息

前方有坑

起初我的错误在于, 我想当然的把登录失败的情况和取消的情况以error的形式传递到了 .onError()中, 用来通知外面. 但是这样一来, 就相当于告诉外面, 事件完成了, 但是对于那些需要一直监听结果的观察者(Observer)来说, 这是一个错误的做法, 因为他们可能只是在初始化的时候, 只进行一次注册监听(类似监听者模式), 如果这时候你以onError()的方式告诉他, “嘿, 我们结束了.” 这显然不合适. 后来, 我换成了使用onNext()事件, 把error传递出去. 这样, 对于外面来说, 事件就没有结束.

第二个坑

这个信号塔有可能被多个监听者监听, 那么, 就有这种可能, 当增加一个监听者的时候, 他就会把所有的事情重复一遍, 我是这样发现这个问题的, 当我发送了一个网络请求的时候, 我会抓到两次甚至多次包

从网上爬了好多资料, 最后忘记在哪里看到了一个人说, 要进行share(), 也就是说, 所有观察者共享一个结果. 当然在上面的代码中, 也可以看到这样的功能点.

创建

有了上面的例子的铺垫, 就可以愉快的看一些概念了.

Empty

发送一个空消息, 也就是立刻回执行到onComplete事件.

Never

永远都不会发送消息, 应用场景很少, 例如某些信号转换的时候, 一些分支不需要继续走下去了, 就可以发送一个never.

Just

发送一个对象, 然后发送一个完成事件, 理解为这样:
onNext(AnyObject)
onComplete()

Create

手动去创建, 这里就可以写一些复杂的逻辑了.

failWith

发送一个onError()事件.

当然, 创建的方式并不限于以上几种, 但是上面的我都用过, 所以敢写出来, 没用过的, 我感觉脱离应用场景说概念, 无非是浪费生命, 而那些没有用到过的, 也不是没有用, 我相信存在的就是合理的, 需要写一段时间代码后, 回来再看, 那时候有的地方你会霍然开朗.

Subject

PublishSubject

在上面的例子中, 使用了PublishSubject, 这相当于一个信号发射塔, 通过asObserver方法转为一个观察者, 向外发射信号, 然后使用share方法来将发射的模式转为共享模式.

Variable

这个在实际用途中, 我使用它来进行一个对象的属性变化.

这一类型的值, 不会发送onError和onComplete, 而是在析构的时候发送onComplete, 这里的例子也是在析构函数中发送哦弄Complete, 虽然因为是单例不会执行, 但是好的编码习惯要有.

Transform

Map

首先是map, 这里的map取映射的意思. 可以将一个事件流转为另一种形式, 而且是对流的所有事件进行转换.
在项目中, 我使用最多的场景是response中获取json

另一个场景, 对返回的结果进行过滤, 包括业务逻辑错误(e.g 对于error code 不为0的进行过滤)

FlatMap

同样带Map, 但是在使用的时候, 完全不一样, 因为维度不同, 所以使用方式不同. 我举个例子, 然后咱们一起看一下区别. 来个简单的先

The difference between map and flatmap

首先从Block的返回值来看, map的返回值, 是对象的本身, 然后回自动包装成Observable, 而flatmap需要返回成品Observable, 所以我说, 他们的维度不同. 当然还有一些不同点, 入坑时间短, 还没有用到, 所以不敢乱BB.

Subscribe

subscribeNext, subscribeCompleted, subscribeError

这三个会触发时间, 订阅对应的事件, 换句话说, 如果你不去订阅一个事件, 它并不会做什么.

doOn

并不会触发事件, 不过在有人触发后, 会走doOn之中的逻辑.

结束

暂时在项目中用到的就是这么多, 总体来说, 开始如果没有人带, 自己摸索, 坑比较多, 有可能陷入多个坑中, 很郁闷, 而且调用堆栈过长, 不利于调试. 优势在于Rx是一种思想, 当我们无法选择夸平台夸语言的时候, 也许可以考虑一下更加抽象的一层, 用思想去统一.
